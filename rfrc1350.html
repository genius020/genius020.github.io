Section 01

Purpose of TFTP

TFTP stands for Trivial File Transfer Protocol. It is a very simple tool for sending files from one computer to another over a network. The word "trivial" tells you everything — it's designed to be as basic as possible.

TFTP works on top of UDP (User Datagram Protocol), which allows it to move files between computers on different networks. Because it uses UDP, it can work across the Internet.

What can TFTP do?

Only two things: read files (download) and write files (upload) from a remote server. That's it. It cannot list directories, and there's no way to log in or verify who you are.

Three modes of transfer are supported: netascii (standard text format), octet (raw binary data — an exact copy of bytes), and mail (sending text to a user instead of a file). However, mail mode is obsolete and should not be used.

💡

Why so limited?

Simplicity is the point. Since TFTP is so small and easy to build, it's perfect for situations like booting up a device that doesn't have an operating system yet — the device can use TFTP to download files it needs to start up.

Section 02

Overview of the Protocol

Every transfer begins with a request — either to read or write a file. If the server agrees, the connection opens and the file is sent in small, fixed blocks of 512 bytes.

Each data block must be acknowledged before the next one can be sent. This creates a step-by-step rhythm: send a block, wait for "got it", send the next block.

When a block arrives that is less than 512 bytes, that signals the transfer is finished.

📖

Think of it like this...

Imagine passing a book to someone one page at a time. You hand over page 1, they say "got it", then you hand over page 2, they say "got it", and so on. If they don't respond after a while, you resend the last page. When you send a page that's shorter than a full page, that means "we're done."

If a packet gets lost in the network, the receiver will timeout and resend its last message, which causes the sender to resend the lost packet. The sender only needs to remember one packet at a time — the lock-step acknowledgment guarantees everything before it was received.

Most errors cause the connection to close. An ERROR packet is sent but never retransmitted, so the other side relies on timeouts to detect the problem.

Section 03

Relation to Other Protocols

TFTP doesn't work alone — it sits on top of other protocols, like layers of a sandwich. When a TFTP packet travels through a network, it gets wrapped in multiple layers of headers.

Local Medium Header e.g., Ethernet — moves packet physically
Internet (IP) Header Routes packet across the internet
Datagram (UDP) Header Handles delivery between programs
TFTP Header + Data The actual file transfer content

Think of it like putting a letter inside multiple envelopes. The innermost envelope is TFTP (your actual content), wrapped in UDP, wrapped in IP, wrapped in whatever your local network uses.

What does TFTP care about?

TFTP ignores the Internet header entirely — that's IP's job. But it does use two things from the UDP header: the source port and the destination port. These ports identify who is talking to whom. TFTP calls them Transfer Identifiers (TIDs), and they must be between 0 and 65,535.

The TFTP header itself is tiny — just 2 bytes containing an opcode that tells you what kind of packet it is (read request, data, error, etc.).

Section 04

Initial Connection Protocol

This is how two computers start talking when one wants to send or receive a file. Think of it as the handshake before the actual transfer begins.

Before the conversation starts, each side picks a random number called a TID (Transfer Identifier). Think of it like each person choosing a random phone number just for this one call.

Example: Writing a File

1
Host A sends a WRQ to Host B

Source = A's TID, Destination = Port 69 (the server's "front door")

2
Host B sends an ACK (block #0) to Host A

Source = B's TID, Destination = A's TID

3
Connection established!

Host A can now start sending DATA packets beginning with block #1. Port 69 is no longer used.

After the connection is set up, both sides always check the TID on every incoming packet. If a packet arrives with a wrong TID, it gets thrown away — it's not part of this conversation. An error is sent back to the wrong sender, but the real connection continues undisturbed.

Why might a wrong TID appear?

If a request packet gets duplicated in the network, the server might respond twice with different TIDs. The first response is accepted, the second is rejected. The original connection keeps going perfectly fine.

Section 05

TFTP Packets

TFTP uses exactly five types of packets. Every communication is one of these:

OpcodeNamePurpose
1RRQRead Request — ask to download a file
2WRQWrite Request — ask to upload a file
3DATAData — carries actual file content
4ACKAcknowledgment — confirms receipt
5ERRORError — reports a problem

RRQ / WRQ Packet (Opcode 1 or 2)

2 bytes Opcode 01 or 02
string Filename e.g. "hello.txt"
1 byte 0 separator
string Mode e.g. "octet"
1 byte 0 end marker

The request tells the server which file you want and what transfer mode to use. Mode can be "netascii", "octet", or "mail" (case insensitive).

DATA Packet (Opcode 3)

2 bytes Opcode 03
2 bytes Block # starts at 1
0–512 bytes Data actual file content

Block numbers start at 1 and increase by one. If the data field is exactly 512 bytes, more blocks follow. If it's 0–511 bytes, the transfer is done.

ACK Packet (Opcode 4)

2 bytes Opcode 04
2 bytes Block # echoes DATA

The simplest packet — it just echoes the block number of the DATA packet it's acknowledging. A WRQ is acknowledged with block number 0.

ERROR Packet (Opcode 5)

2 bytes Opcode 05
2 bytes ErrorCode 0–7
string ErrMsg human-readable
1 byte 0 end

Here are all the error codes defined in the protocol:

CodeMeaning
0Not defined, see error message
1File not found
2Access violation
3Disk full or allocation exceeded
4Illegal TFTP operation
5Unknown transfer ID
6File already exists
7No such user
Section 06

Understanding Null Bytes

In the RRQ/WRQ packets, you'll notice those 1-byte zeros between the fields. These are called null terminators. Their job is to act as separators — they tell the computer "this string has ended here."

The filename and mode are both variable length — they can be any size. Without the zero bytes, the computer would see everything smashed together with no way to tell where one field ends and the next begins.

Real Example Reading "hello.txt" in octet mode:
01 | hello.txt | 0 | octet | 0

First 0 → "filename ended here"
Second 0 → "mode string ended here"
🧱

Think of them as walls

The zero bytes are like walls between rooms. Without them, all the rooms merge into one big space and you can't tell where the kitchen ends and the bedroom starts.

Section 07

Normal Termination

This is how a transfer ends when everything goes well.

The key rule: when a DATA packet arrives with less than 512 bytes (anything from 0 to 511), that means it's the last packet. The transfer is complete.

How Normal Termination Works

1
Sender sends final DATA packet

This packet contains less than 512 bytes of data

2
Receiver sends back a final ACK

Acknowledges the last data block

3
Receiver waits a bit before closing

This is called "dallying" — in case the ACK got lost and needs to be resent

4
Connection closed

Both sides are done. Transfer successful!

Why wait after the final ACK?

If the final ACK gets lost, the sender will think the last DATA packet was lost and resend it. If the receiver has already closed the connection, nobody is there to respond. So the receiver hangs around briefly just in case.

From the sender's side, after sending the last DATA packet, it keeps resending until either an ACK comes back (success!) or it gives up after too many timeouts. Even if the sender times out, the transfer might still have been successful — the receiver may have gotten the data, but the ACK got lost.

Section 08

Premature Termination

This is the ugly ending — when something breaks during the transfer.

When an error occurs (file doesn't exist, disk gets full, something unexpected happens), an ERROR packet (opcode 5) is sent and the connection is shut down.

The catch

The ERROR packet is just a courtesy. It is never retransmitted and never acknowledged. The sender fires it off once and walks away. This means the other side might never receive it.

So how does the other side figure out the connection is dead? Through timeouts. If it keeps waiting for a response and nothing comes, it eventually gives up and assumes something went wrong.

📞

Comparing the two endings

Normal termination is like ending a phone call properly — "Okay, bye!" "Bye!" Click. Premature termination is like the call suddenly dropping — one side sends a quick "something went wrong" but doesn't wait to see if the other person heard it. Eventually the silence tells them the call is over.
